基础概念扫盲

进门前先了解下基本概念

概念 描述 作用
Controller 控制器 处理路由、解析请求参数
Handler 处理器 控制器内处理路由的方式
Service 服务层 实现业务逻辑（业务逻辑、操作数据库）
DTO 数据传输对象 Data Transfer Object 控制请求体中的数据对象
Module 模块 项目模块化（用户模块、支付模块）
Entity 实体 映射数据库中表的字段
Ioc 反转控制（依赖注入） Inverse Of Controller 声明依赖后，nest会自动注入依赖的实例
Aop 面向切面编程 Aspect Oriented Programming 在多个请求和响应中插入可复用的逻辑，比如日志记录、请求拦截、响应拦截等，包含：middleware、interceotor、guard、exception、filter、pipe等

全局模块

模块导出provider（即使用@Injectable装饰的类），另外一个模块需要在模块中imports，才可以使用别的模块的provider。

  1. A模块将AService导出（导出的是Aservice）；

  2. B模块将AModule导入（导入的是整个A模块）；

如果被导出的模块，在很多模块中都被使用到，需要一个个模块中imports，是一件很繁琐的事情！

所以可以将公共的模块，使用@Global装饰器设置成全局模块。（全局模块缺点：不好追溯provider，降低代码可维护性）

IOC

Nest实现了一套依赖注入机制，叫做IoC（Inverse of Control 反转控制）。

在@Module中声明了依赖，不需要手动创建依赖，Nest的IoC容器会自动创建依赖并且注入依赖（注入通过@Inject声明依赖）。

在@Module里的providers数组里，就是声明需要注入的依赖对象，这里叫做providers。

providers有很多中写法：

  1. 默认的XxxxService，等同于是useClass
  2. useValue
  3. useFactory

注入的方式也分为（两种效果都一样）：

  1. 属性注入，通过@Inject
  2. 构造器注入，（private readonly xxxService：XxxServie）

AOP

Aspect Oriented Programming 面向切面编程；

前端发起请求到后端接收：经过controller（参数接收）、service（逻辑处理）、repository（数据库访问）；

AOP的作用就是在controller（接收参数之前）的链路中，透明的加上一些任务，例如：日志记录、权限控制、异常处理等；

这样处理既可以保持业务逻辑的纯粹性，逻辑又可以复用，还可以动态的增删；

Pipe管道

Pipe作用是在参数传给Get方法的 handler 之前对参数做一些验证。

内置的管道有以下这些：

内置管道 作用 使用场景
ValidationPipe 用于验证传入的数据是否符合指定的规则。它依赖于 class-validator 和 class-transformer 库，通过在数据传输对象（DTO）上使用装饰器来定义验证规则 适用于对请求体、查询参数等进行复杂的数据验证，确保数据的完整性和合法性
ParseIntPipe 将传入的字符串参数转换为整数。如果转换失败（例如输入不是有效的数字），会抛出异常 常用于处理路径参数或查询参数中的数字类型数据，例如获取用户 ID 或分页参数
ParseBoolPipe 将传入的字符串转换为布尔值。通常会将字符串 "true" 转换为 true，将 "false" 转换为 false，其他值可能会导致异常 用于处理布尔类型的查询参数或路径参数，例如启用/禁用某个功能的开关
ParseArrayPipe 将传入的字符串数组或以某种格式传递的数组数据解析为 JavaScript 数组 适用于处理以查询字符串形式传递的数组数据，例如批量获取资源的 ID 列表
ParseUUIDPipe 验证传入的字符串是否是有效的 UUID（通用唯一标识符）。如果不是有效的 UUID，会抛出异常 用于处理需要 UUID 格式的路径参数或查询参数，例如资源的唯一标识符
DefaultValuePipe 当传入的参数为 undefined 或 null 时，为参数提供一个默认值 适用于某些参数可选的情况，当参数未提供时，可以使用默认值以避免错误
ParseEnumPipe 将传入的字符串转换为枚举类型中的成员。如果输入的值不在枚举中，会抛出异常 用于处理需要枚举类型参数的场景，例如根据枚举值执行不同的逻辑
ParseFloatPipe 将传入的字符串转换为浮点数。如果转换失败，会抛出异常 适用于处理需要浮点数类型的参数，例如价格、评分等
ParseFilePipe 用于处理上传的文件。它可以验证文件的类型、大小等属性，确保上传的文件符合预期 适用于文件上传接口，用于验证和处理上传的文件数据

Docker

初识

后端系统会部署很多服务，包括我们自己开发的http服务，还有 mysql、redis 等中间件的服务，部署它们需要一系列依赖的安装、环境变量的设置，如果换一台服务器，意味着又要重复一遍环境搭建、依赖安装的操作。

就很麻烦。

而 Docker 就能解决这个问题：

它把系统的所有文件封装成一个镜像，每个镜像运行在单个容器里，每个容器都有独立的操作系统环境，比如文件系统、网络端口等，一台机器上又可以跑多个容器。

Docker 提供了 Docker Hub 镜像仓库，可以把本地镜像 push 到仓库或者从仓库 pull 镜像到本地。

镜像制作

编写Dockerfile

docker 容器内就是一个独立的系统环境，想想如果在这样一个系统上，要安装 nginx 服务，怎么做呢？

需要执行一些命令、复制一些文件进来，然后启动服务。

制作镜像自然也要进行这样的过程，不过可以自动化。

只要在 dockerfile 里声明要做哪些事情，docker build 的时候就会根据这个 dockerfile 来自动化构建出一个镜像来。

例如：

逐行解释

FROM node:latest

这行指定了基础镜像。它告诉 Docker 使用官方的 Node.js 镜像，并且使用最新的版本。这意味着容器将包含 Node.js 的运行环境，包括 Node.js 和 npm。

WORKDIR /app

这行设置了工作目录。在容器中，所有后续的命令（如 COPY、RUN 等）都会在这个目录下执行。这里将工作目录设置为 /app。

COPY . .

这行将当前目录（构建上下文目录）中的所有文件复制到容器的 /app 目录中，**.** 表示当前目录，第一个 . 是源路径，第二个 . 是目标路径

RUN

在容器内执行命令

EXPOSE 8080

声明了容器将监听的端口，这里是 8080。虽然这不会实际映射端口，但它会告诉 Docker 这个容器将使用哪个端口提供服务

CMD [“http-server”, “-p”, “8080”]

指定了容器启动时运行的默认命令。它会启动 http-server，并将其监听的端口设置为 8080

在 Dockerfile中指定 VOLUME 和 启动镜像的时候 设置的 VOLUME 有什么不同呢

在 dockerfile 里指定 VOLUME 之后，如果你 docker run 的时候没有带 -v，那会放在一个临时的目录里，docker会随机生成一个目录作为数据卷挂载上去，如果误删了这个容器，也可以在这个随机目录中，找回数据。

设想下，如果你跑了个mysql容器，存了很多数据，但是跑容器的时候没有指定数据卷，有一天误删了mysql容器，那所有的数据都找不回了!!!

为了避免这个情况，mysql的Dockerfile必须声明VOLUME，就算在启动镜像的时候，没有指定-v，将来也可以在数据卷中找回数据。
