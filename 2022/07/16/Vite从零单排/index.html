Vite从零单排

发表于2022-07-16|更新于2023-04-11|vite

什么是构建工具

浏览器只认识html,js,css的代码

而我们日常开发项目，有可能用到了 TypeScript/React/Vue/less/sass/语法降级/体积优化等。

TypeScript ==> 使用 tsc 将 typescript 代码转换为 js 代码。

React/Vue ==> 安装 react-complier/vue-complier 将对应文件转换为render函数。

less/sass ==> 安装 less/sass-loader 等编译工具将样式编译为 css 文件。

语法降级 ==> 将更高级的 js 语法转换为旧浏览器认识的语法。

体积优化 ==> uglifyjs 讲代码进行压缩变成体积更小、性能更高的文件。

然后我日常的开发的项目，最终的产物都是 html、css、js。

所以构建工具在这个过程，帮我们做了哪些脏活烂活？

与 webpack 的相爱相杀

webpack 支持多模块语法：比如 Commonjs 模块化、Esmodule 模块化，侧重的是兼容性

Vite 只支持 Esmodule 模块化，更加侧重浏览器端的开发体验。

灵魂一问：Vite 会不会把 webpack 干翻？ 其实是不会的。两者的侧重点不一样。

webpack 的打包流程是将所有模块打包编译好之后，才启动开发服务器。

所以项目越大，代码越多，解析的就越慢。

vite 比较粗暴，直接启动开发服务器，同时加载入口文件entry，然后根据不同模块按需加载。

无论项目多大、代码多臃肿，都没关系。

你要理解 vite 脚手架和 vite

vite 中文官网

如果我们再全局执行yarn create vite，内部会执行以下操作:

  1. 帮我们全局安装脚手架：create-vite(注意：并不是安装的 vite)，这里面帮我们预设了很多东西，就好像 vue-cli 的功能。
  2. 直接运行 create-vite bin 目录下的一个执行配置

说白了就是 vite 的模版。

初体验

产生报错

在普通的 js 文件中，引入 lodash 进来，在浏览器中打开会报错

浏览器提示我们在导入这个 lodash 的时候，必须使用/,./或者../(要么是相对路径要么是绝对路径)，原因很简单：因为浏览器并不知道 node_module 目录，不知道 lodash 是什么东西。

灵魂质问：

如果 es 官方在我们导入非绝对路径、非相对路径的资源的时候，默认帮我们去 node_module 检索一遍模块，会发生什么劲爆的事情？

：假如你这个包又引入了其他的包，其它包又引入了其他包（套娃下去）。这样一来，浏览器在加载模块的时候，会依次跟随依赖包加载下去，这样一来非常的占用网络资源，所以直接读取本地模块 node_module 才是最佳实践。

解决报错

现在在项目中依赖上 vite

添加启动命令

通过命令启动项目

此时项目里能正常打印lodash，这就是 vite 帮我们做的事情，所有模块加载、语法降级等功能已经帮我们集成，真正的开箱即用！

vite 在构建过程中，碰到类似于非相对路径、非绝对路径的模块引用，会尝试开启路径补全。

vite 依赖预构建

依赖包表现

找寻依赖的过程：从自身逐级向上查找，直到搜寻到根目录或者寻找到对应的依赖为止。

生产环境和开发环境的依赖预构建：

开发环境：每次依赖预构建所重新构建的相对路径都是正确的

生产环境：vite 会全权交给 rollup 的库去完成生产环境的打包

起因

假如有些依赖包使用的是commonJs规范导出的方式，vite 是不认识的，前面提到过 vite 只认识ESmodule规范，这时候就有了依赖预构建的功能。

依赖预构建：

vite 在构建过程中，先找到对应的依赖，然后调用esbuild(对 js 语法进行处理的一个库)，将其他模块化规范转换成ESmodule规范，然后放到当前目录下的node_module/.vite/deps

结果

  1. 解决了不同的第三方包会有不同的导出规范的问题，这个是 vite 没法约束的。
  2. 对于依赖包内的模块引用，使用路径重写解决问题。
  3. 网络多包传输的性能问题（后面再补充），有了依赖预构建以后无论他有多少额外的 export 和 import，vite 都会尽可能的将他们进行集成，最后产物只有一个或者几个模块。

附带知识点：

vite 配置文件语法提示

vite 在构建的时候，会自动读取项目中的vite.config.js配置文件，通常这个文件是导出一个对象。

语法提示

如果想要有语法提示：

  1. 使用 webstorm 会有语法补全

  2. 使用 vite 导出的defineConfig模块，此时输入optimizeDeps就有了语法提示

  3. 使用@type注释的方法

环境配置

区分开发还是生产环境，使用command:'build'|'serve'

过去 webpack 环境是通过文件名字来区分的，webpack-dev-config、webpack-prod.config

使用 vite 的区分：

  1. 添加 3 个文件vite.base.config.js、vite.dev.config.js、vite.prod.config.js分别代表基础配置文件、开发环境配置文件、生产环境配置文件，三个文件分别导出各自所需的对象。

  2. 在vite.config.js文件中，导入 3 个配置文件，通过策略模式映射出当前所需环境的配置。

  3. 最后导出所需要的配置对象。

command 是通过什么东西来区分 build,serve？

：是 build 还是 serve 主要取决于启动命令来决定 yarn build 还是 yarn dev

附带知识点

为什么函数注释要写@returns @param？

：当给一个函数添加了返回类型之后，函数的返回值就会有语法提示！

vite 环境变量

根据当前代码环境动态变化的值叫做环境变量。

服务端环境变量处理

在 vite 中对环境变量的处理：使用dotenv第三方库，这个库会自动读取.env文件，将文件中的环境变量映射成 key value 的形式，再注入到process对象中。

APP_KEY = 123 映射成 [key,value] = [‘APP_KEY’, 123]

vite 默认不加载.env文件，因为这些文件需要在执行完 Vite 配置后才能确定加载哪一个，以下两个配置项会影响加载行为。

  • root
  • enDir：用来配置当前环境变量的地址

所以当你确认需要加载时，可以通过loadEnv来确认 env 文件，loadEnv 接受 3 个参数：

mode: env 文件后面的名字，例如.env.development，这里就是 development

envDir：env 文件的绝对路径，一般是process.cwd()

profixes：env 文件内环境变量名的前缀，默认是VITE_

项目中会通过三个 env 文件来区分：

  • .env：所有环境下都需要用到的环境变量
  • .env.development：开发环境需要用到的环境变量
  • .env.production：生产环境需要用到的环境变量

客户端环境变量处理

在客户端使用import.meta.env来读取环境变量，但是 vite 为了防止我们将隐私的变量直接送进import.meta.env，所以在客户端使用的环境变量需要使用VITE_前缀。

如果想在客户端修改环境变量的前缀，需要使用envPrefix，类型接受 string 或者 string[]

envPrefix 不能设置为空字符串，vite 会报错

vite 如何让浏览器解析.vue 文件

浏览器是如何渲染.vue 文件？

：浏览器只认识 html、css、javascript 代码，但是浏览器的 network 中可以看到，例如 vue 工程项目也可以读取.vue 文件。

其实这是因为 vite 在遇到.vue 结尾的文件的时候，将里面的代码转换成 render 函数，浏览器这里的App.vue是个文件名，但是里面的内容已经被转换成 js 代码。

现在动手使用koa简单实现下这个功能：

先模拟出 vue 工程项目的项目结构

再创建一个 index.js，模拟服务端的编译过程

通过匹配对应的文件地址，来决定将Content-Type(text/html 、text/javascript)渲染成什么类型的文件，再让浏览器加载。

vite 对 css 模块的处理

默认模块处理

vite 默认对 css 文件是可以直接渲染的。

大概步骤和上面提到的 vite 解析.vue 文件 的原理差不多

  • vite 加载到.css 文件的时候，用fs模块读取 css 文件内容
  • 直接创建一个 style 标签，将 index.css 中文件内容直接 copy 进 style 标签里
  • 再将 style 标签插入到 index.html 中
  • 将 css 文件中的内容替换成 js 脚本，让浏览器去执行这个文件。

cssModule

假设现在在协同开发，同事 A 有一个样式文件，同事 B 有一个样式文件，假如两个人的样式类目起冲突，被编译到 style 标签中，后者的类名样式会覆盖前者。

为了避免这个冲突，所以有了 css module，它表示需要开启 css 模块化：

componentA.module.css

  1. module.css 是一种约定，表示需要开启 css 模块化
  2. 它会将你所有类名进行一定规则的替换（将footer替换成_footer_hash值的格式）
  3. 同时创建一个映射对象{ footer: “_footer_i22st_1” }
  4. 再将替换过后的内容塞进 style 标签中
  5. 将 component.module.css 的内容替换成 js 脚本
  6. 将创建的映射对象在脚本中默认导出
